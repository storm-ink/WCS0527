# WCS 瘦身与 B/S 架构改造计划

## 1. 背景

当前 WCS 仓库同时存在以下多套形态：

- 以 `WCS.APP` 为代表的 C/S 主客户端
- 以 `Wcs.Client.App` 为代表的独立轻客户端
- 以 `WebUI`、`Wcs.WebClient` 为代表的旧版 Web 表现层
- 以 `WcfServices/*` 为代表的 WCF 服务层
- 以 `WEBAPI`、`Wcs.WebApi`、`ZHQXC/WebAPI` 为代表的多套 API 层
- 以 `RemoteDiagnose/*`、模拟程序、测试插件为代表的诊断/测试/历史遗留工程

这导致项目边界不清、入口重复、部署面过大、维护成本高，也不利于后续统一向 B/S 架构演进。

本计划目标是：

1. 对现有 WCS 工程进行分层收口，保留后端核心处理能力。
2. 摒弃以桌面客户端为中心的 C/S 运行形态，改造为浏览器前端 + 常驻后端服务的 B/S 架构。
3. 在保证现场可运行的前提下，逐步摘除未使用、低价值、重复的项目。

## 2. 当前架构判断

### 2.1 当前生产形态

当前生产形态本质上不是“多个独立服务协同”，而是：

- 一个主宿主进程负责启动框架、读取配置、拉起启动项
- 核心调度、设备通信、任务拆分、业务处理大多驻留在同一运行域
- 插件通过动态扫描的方式挂载到主程序
- 部分能力通过 WCF 或 WebAPI 对外暴露

因此，后续 B/S 改造不能简单理解为“把 WinForm 页面搬到 Web 页面”，而应理解为：

> 将现有 WCS 核心引擎服务化，再通过统一 API 为浏览器前端提供能力。

### 2.2 当前核心层级

建议将当前仓库理解为以下层次：

1. **核心基础层**
   - `v1.2/Wcs`
   - `v1.2/Wcs.Framework`
   - `v1.2/Wcs.FrameworkExtend`

2. **设备实现层**
   - `v1.2/Wcs.DefaultImplementCollection.*`
   - `v1.2/BaseCommunication/S7`

3. **现场业务层**
   - `ZHQXC`
   - `ZHQXC/处理程序`
   - `ZHQXC/启动程序`
   - `ZHQXC/路径策略`
   - `ZHQXC/WebAPI`

4. **宿主与表现层**
   - `v1.2/WCS.DEVAPP`
   - `v1.2/WcsConsoleApplication`
   - `v1.2/Client/*`
   - `v1.2/Plugins/*`
   - `v1.2/WebUI`
   - `v1.2/Wcs.WebClient`

5. **对外接口层**
   - `v1.2/WcfServices/*`
   - `v1.2/WMSServices`
   - `v1.2/WEBAPI`
   - `v1.2/Wcs.WebApi`

6. **诊断/模拟/测试层**
   - `v1.2/RemoteDiagnose/*`
   - `v1.2/OnlineLogDiagnosis`
   - `v1.2/Plugins/Debugger`
   - `v1.2/Plugins/CraneTest`
   - 模拟程序、测试工程、独立工具

## 3. 目标架构

## 3.1 目标形态

目标架构采用四层：

1. **WCS Core Engine**
   - 保留现有设备通信、任务调度、路由、事件处理、任务状态管理能力
   - 保持配置驱动和现场扩展能力

2. **Host Service**
   - 新的常驻后端宿主
   - 负责系统启动、配置加载、任务线程、设备连接、事件处理
   - 生产上替代 `WCS.APP` 作为主运行载体

3. **Unified API**
   - 面向浏览器前端和外围系统的统一接口层
   - 承接当前 WCF、旧 WebAPI、客户端调用接口的能力

4. **BS Frontend**
   - 统一的 Web 前端
   - 承载监控、任务查询、设备状态、人工干预、报警、统计等页面

## 3.2 核心原则

1. **核心引擎保留，表现层重做**
   - 不重写设备通信和调度核心
   - 优先重构宿主与接口层

2. **先收口，再迁移，再删除**
   - 任何项目在没有运行证据前，不做物理删除

3. **运行链路优先**
   - 以现场运行链路为准，而不是仅以项目引用关系为准

4. **统一入口**
   - 对内统一宿主
   - 对外统一 API
   - 对用户统一 Web 前端

## 5. 分阶段实施路线

## Phase 0：现状盘点与运行证据确认

### 目标

明确哪些工程真正参与生产运行，哪些只是历史遗留或辅助工具。

### 工作项

1. 整理解决方案、工程、插件、WCF、API、工具清单
2. 交叉核对以下信息：
   - `startups.xml`
   - `services` 配置
   - 部署目录中的 DLL
   - 现场实际启用的页面与接口
3. 输出“项目去留矩阵”

### 输出物

- `保留 / 迁移 / 下线 / 待确认` 清单
- 最小运行链路说明

### 验收标准

- 每个候选删除项都有明确结论与依据

## Phase 1：构建最小可运行内核

### 目标

从当前“大而全”方案收口到“生产最小运行集”。

### 工作项

1. 建立新的最小解决方案，仅包含：
   - 核心框架
   - 现场业务
   - 必要设备实现
   - 必要集成能力
2. 从解决方案级别移除明显非生产工程
3. 保持现有运行结果不变

### 输出物

- `WCS 最小运行解决方案`
- 最小部署清单

### 验收标准

- 在不依赖桌面主界面的情况下，核心引擎可独立启动

## Phase 2：核心宿主服务化

### 目标

将核心引擎从桌面主程序中解耦，形成可常驻运行的后端宿主。

### 工作项

1. 以 `WcsConsoleApplication` 为基础演化新的服务宿主
2. 将 `WCS.APP` 中与 UI 无关的启动逻辑下沉
3. 明确宿主生命周期管理：
   - 启动
   - 停止
   - 重连
   - 日志
   - 健康检查

### 输出物

- 新宿主进程
- 宿主启动说明

### 验收标准

- 不启动 WinForm 也能完整拉起设备通信和调度能力

## Phase 3：统一接口层建设

### 目标

将 WCF、旧 API、客户端服务接口收口为统一 API。

### 工作项

1. 盘点现有接口能力：
   - 任务查询
   - 设备状态
   - 报警查询
   - 人工干预
   - WMS 调用
   - AGV 调用
2. 设计统一 API 契约
3. 用 API 替代旧 WCF/客户端调用通道
4. 保留必要兼容期

### 输出物

- API 清单
- 统一接口定义
- 替代关系表

### 验收标准

- 浏览器前端与外围系统均通过统一 API 访问 WCS 能力

## Phase 4：Web 前端建设

### 目标

逐步替代桌面客户端和桌面插件的操作/监控页面。

### 优先迁移页面

1. 首页总览
2. 设备监控
3. 任务列表
4. 手工任务
5. 报警与消息
6. 报表统计

### 暂缓页面

1. 低频调试工具
2. 现场专用测试窗体
3. 复杂工程化维护页面

### 输出物

- Web 前端一期
- 页面与原插件功能映射表

### 验收标准

- 日常运维操作可完全在浏览器完成

## Phase 5：旧项目下线

### 目标

在新宿主、新接口、新前端稳定后，逐步删除旧项目。

### 下线顺序建议

1. `Client/*`
2. `WebUI`
3. `Wcs.WebClient`
4. `WcfServices/*`
5. `WCS.DEVAPP`
6. `RemoteDiagnose/*`
7. 诊断/测试/模拟工具

### 验收标准

- 被下线项目功能已有替代实现
- 部署清单已同步收敛
- 现场运行链路不再依赖旧工程

## 6. 近期实施建议（建议优先做）

建议下一阶段先做以下 6 件事：

1. 输出一版正式的**项目去留矩阵**
2. 输出一版正式的**最小部署清单**
3. 明确当前生产必须插件与必须启动项
4. 建立新的**最小运行解决方案**
5. 设计统一 API 的模块边界
6. 确定 Web 前端一期范围

## 7. 风险与控制

### 风险 1：动态加载导致误删

当前系统存在启动项配置、服务配置、插件扫描，不能只看代码引用。

**控制措施：**

- 删除前必须做运行证据核对
- 删除优先从解决方案剔除开始，不直接删源码

### 风险 2：桌面插件中混有业务逻辑

部分插件不仅是界面，还可能携带业务逻辑。

**控制措施：**

- 先识别插件中的业务逻辑
- 先下沉到应用服务层，再迁移前端

### 风险 3：对外围系统接口影响大

WMS、AGV、其他系统可能依赖现有 WCF 或旧 API。

**控制措施：**

- 新旧接口并行一段过渡期
- 提前形成接口替代关系表

### 风险 4：项目收口后编译链变化

多个老项目共存时，解决方案清理后可能暴露隐藏依赖。

**控制措施：**

- 先做最小集编译验证
- 每轮只收敛一批项目

## 8. 计划结论

本次 WCS 优化的主线应确定为：

> **保留后端核心处理能力，统一宿主与接口层，逐步以 Web 前端替代桌面客户端，最后摘除历史遗留和重复项目。**

不建议一开始就大规模物理删除工程。正确顺序应为：

1. 盘点
2. 收口
3. 服务化
4. API 统一
5. 前端替换
6. 旧项目下线
